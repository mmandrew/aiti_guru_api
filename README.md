# Сущности. 
В рамках этого проекта я буду работать с 3 сущностями: пользователи, заказы и платежи

* Один пользователь может иметь ноль, один или несколько заказов.  
* Каждый заказ пользователь может оплачивать 1 или несколькими платежами.  
* Однако заказы могут висеть и неоплаченными.  
* Пользователь не может тратить больше, чем у него есть на счете. Также он не может переплатить за заказ.  
* Внесенные платежи могут быть отменены. Тогда деньги возвращаются пользователю на счет, а информация о платеже удаляется.

# Работа с СУБД.
В качестве СУБД я выбрал PostgreSQL. Я создал схему aiti_guru_api и в ней 3 таблицы для хранения данных.

## Таблица users

```sql
CREATE TABLE aiti_guru_api.users (
	id serial4 NOT NULL,
	"name" varchar NOT NULL,
	current_money float8 NOT NULL,
	CONSTRAINT current_money_ge_0 CHECK ((current_money >= (0)::double precision)),
	CONSTRAINT users_pkey PRIMARY KEY (id)
);
```

* `id` - уникальный номер пользователя.   
Задается автоматически при добавлении нового пользователя.
* `name` - имя пользователя.
* `current_money` - текущая сумма на счете пользователя.  
Он может тратить их на оплату заказа.
    В случае оплаты соответствующая сумма списывается со счета. Значение не может быть меньше 0.

## Таблица orders

```sql
CREATE TABLE aiti_guru_api.orders (
	id serial4 NOT NULL PRIMARY KEY,
	user_id int4 NOT NULL REFERENCES aiti_guru_api.users(id),
	item_name varchar NULL,
	"cost" float8 NOT NULL,
	remained_cost float8 NOT NULL,
	status varchar NOT NULL,
	created_at timestamp DEFAULT now(),
	CONSTRAINT remained_cost_ge_0 CHECK ((remained_cost >= (0)::double precision)),
	CONSTRAINT remained_cost_no_more_than_cost CHECK ((remained_cost <= cost)),
	CONSTRAINT status_set CHECK (((status)::text = ANY ((ARRAY['Не оплачен'::character varying, 'Частично оплачен'::character varying, 'Оплачен'::character varying])::text[])))
);
```

* `id` - уникальный номер заказа.  
  Задается автоматически при добавлении нового заказа. (Хотя по тз новые заказы не добавляются)
* `user_id` - номер заказчика, ссылается на номер пользователя из таблицы users
* `item_name` - название товара
* `cost` - стоимость заказа. Неотрицательная, не меняется на протяжении жизненного цикла заказа.
* `remained_cost` - сумма, которую осталось заплатить.  
  Изначально равна сумме заказа cost. Может варьироваться от 0 до значения cost. 
	При поступлении платежа уменьшается на величину платежа. При создании заказа автоматически определяется в триггере равной cost.
* `status` - принимает одно из трех значений 
  * `Оплачен` - соответствует ситуации, когда cost = remained_cost. Автоматически задается в триггере при создании. 
          Бывает, когда не было еще ни одного платежа или все платежи были отменены и возвращены владельцу заказа.
  * `Частично оплачен` - соответствует ситуации, когда 0 < remained_cost < cost.
  * `Оплачен` - Соответствует ситуации, когда remained_cost = 0.\
Значение поля корректируется триггером в зависимости от remained_cost и cost. Код триггера в файле DDL_main
* `created_at` - дата и время создания заказа, определяется автоматически при создании в триггере.

Триггер, которые задает remained_cost, status и created_at с соответствующей функцией будет в DDL_main


## Таблица payments

```sql
CREATE TABLE aiti_guru_api.payments(
	id serial PRIMARY KEY,
	order_id int NOT NULL REFERENCES aiti_guru_api.orders(id),
	amount float NOT null, 
	type varchar,
	created_at timestamp DEFAULT now(),
	CONSTRAINT type_set CHECK (((type)::text = ANY ((ARRAY[NULL::character varying, 'Наличные'::character varying, 'Эквайринг'::character varying])::text[])))
);
```

* `id` - уникальный номер платежа. Задается автоматически при создании.
* `order_id` - номер заказа, по которому вносится платеж.  
Ссылается на номер заказа из таблицы orders.
* `amount` - сумма платежа.   
Должны соблюдаться 2 условия для успешной регистрации платежа:
  * `0 < amount <= remained_cost` соответствующего заказа.
  * `amount` <= `current_money` пользователя  
Условия проверяются в триггере, код которого в DDL_main.
Тогда строка с платежом запишется в базу, а `current_money` и `remained_cost` должны уменьшаться на значение `amount`. 
Изменения происходят в том же, триггере, который проверят условия для amount.
* `type` - тип платежа.  
Может быть `null`, `Наличные` или `Эквайринг`. В рамках тз я даю возможность записать `null` на случай, если по какой-то причине type неизвестен. 
же не очень ясно, какие операции совершаются с использованием этого параметра.
* `created_at` - дата и время совершения платежа, определяется автоматически

Возврат платежа делается через удаление соответствующей строки из таблицы `payments`. Также значения `current_money` пользователя и `remained_cost` заказа должны увеличиться на сумму платежа.
За изменения этих 2 значений также отвечает триггер, код которого также будет в DDL_main.

По ТЗ должен быть еще столбец, который хранит статус платежа, но по тому же тз я не увидел, от чего этот статус зависит, так что в итоговом решении статус платежа отсутствует.

Столько триггеров и констрейнтов реализовано на случай, если данные будут обновлятся в обход этого API, например, руками в базе.

# Методы, реализованные в API
Весь код методов в routes.py

## POST `/acquiring-start/`
Предназначен для регистрации и обработки платежа. Принимает JSON со следующими данными:
* `order_id` — номер заказа
* `amount` — сумма вносимого платежа
* `type` (опционально, по умолчанию `null`) — тип платежа: наличные или эквайринг

Возвращает:
* В случае успеха: код `200` и `id` сделанного платежа
* В случае неудачи: код ошибки и текст ошибки

Возможные ошибки:
* `404` — если платеж или пользователь не найдены
* `402` — если попытаться внести больше, чем есть на счете или больше, чем необходимо
* `500` — в случае серверной ошибки

Также в случае успеха:
* добавляет запись в таблицу `payments`
* корректирует `current_money` в таблице `users`
* корректирует `remained_cost` в таблице `orders`
* корректирует `status` в таблице `orders`


## DELETE `/refund-acquiring/`
Предназначен для отмены уже совершенного платежа.

Принимает:
- `id` — идентификатор платежа

Возвращает:
- **В случае успеха**: код `200` и JSON с подробной информацией  
  (в рамках тестового задания возвращается подробный ответ для ясности; в продакшене возможно использование `204 No Content`):
  - `success`: `true`
  - `payment_id`: ID отменённого платежа
  - `refunded_amount`: сумма возвращённых средств
  - `user_balance`: баланс пользователя после возврата
  - `order_remaining`: оставшаяся сумма к оплате по заказу
  - `order_status`: обновлённый статус заказа

- **В случае неудачи**: код ошибки и текст ошибки:
  - `404` — если платеж, заказ или пользователь не найдены
  - `500` — в случае серверной ошибки

## GET `/acquiring-check/`

Предназначен для проверки статуса платежа по его идентификатору.

**Принимает:**  
- `id` — идентификатор платежа (обязательный параметр в URL или query-параметре)

**Возвращает:**  
- **В случае успеха**: код `200` и JSON с детальной информацией о платеже:  
  - `id`: ID платежа  
  - `amount`: сумма платежа (в валюте заказа)  
  - `order_id`: идентификатор связанного заказа  
  - `created_at`: дата и время регистрации платежа (ISO 8601 формат)  

- **В случае неудачи**:  
  - `404 Not Found` — если платеж не найден  
  - `500 Internal Server Error` — в случае внутренней ошибки сервера
